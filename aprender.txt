4. MÓDULO DE COMUNICACIÓN (classComunicacion.cs)
================================================================================

Archivo: classComunicacion.cs

Este es el módulo más importante, ya que gestiona toda la comunicación serial.

4.1 Variables clave:
---------------------
- sPuerto: Objeto SerialPort para la comunicación
- tramaLibre (volatile bool): Bandera para controlar el envío de mensajes (evita colisiones)
- tramaRecepcionMensaje: Buffer de 1024 bytes para recibir tramas
- tramaCabacera: Buffer de 5 bytes para la cabecera de las tramas

4.2 Estructura de la Trama (1024 bytes total):
-------------------------------------------------
Byte 0:      Tipo de trama ('M' para mensaje, 'A' para archivo, 'I' para info)
Bytes 1-4:   Longitud del mensaje (4 dígitos, ejemplo: "0123")
Bytes 5-1023: Datos del mensaje + relleno (padding con valor 64 '@')

Total: 1 (tipo) + 4 (longitud) + 1019 (datos) = 1024 bytes

5. FUNCIONES MÁS IMPORTANTES
================================================================================

5.1 InicializaPuerto() - classComunicacion.cs:40-62
-------------------------------------------------------
- Propósito: Configurar y abrir el puerto serial
- Parámetros: nombreP (nombre del puerto COM), velocidad (baudios)
- Configuración del puerto:
  - 8 bits de datos
  - 2 bits de parada
  - Paridad impar (Odd)
  - Tamaño de buffers: 8192 bytes (lectura y escritura)
- Retorno: true si la conexión fue exitosa

5.2 enviarMensaje() - classComunicacion.cs:161-184
------------------------------------------------------
- Propósito: Preparar y enviar un mensaje por el puerto serial
- Pasos:
  1. Espera a que la bandera tramaLibre esté en true (buffer libre)
  2. Convierte el texto a bytes UTF-8
  3. Limita el mensaje a 1019 bytes (máximo payload)
  4. Formatea la longitud como 4 dígitos (ToString("D4"))
  5. Crea la cabecera: "M" + longitud (5 bytes)
  6. Inicia un hilo para enviar la trama (evita bloquear la UI)

5.3 enviandoMensaje() - classComunicacion.cs:186-207
--------------------------------------------------------
- Propósito: Envía físicamente la trama por el puerto serial
- Pasos:
  1. Envía la cabecera (5 bytes)
  2. Envía los datos del mensaje
  3. Envía padding para completar 1019 bytes de payload
  4. Finalmente libera la bandera tramaLibre (en finally para asegurar que siempre se libere)

5.4 SPuerto_DataReceived() - classComunicacion.cs:81-112
------------------------------------------------------------
- Propósito: Evento que se dispara cuando hay datos disponibles en el puerto
- Pasos:
  1. Verifica que haya al menos 1024 bytes para leer (una trama completa)
  2. Lee la trama completa en tramaRecepcionMensaje
  3. Analiza el primer byte para saber el tipo de trama
  4. Si es tipo 'M' (mensaje), inicia un hilo para procesarlo
  5. Si es un tipo no reconocido, muestra alerta y limpia el buffer

5.5 RecibiendoMensaje() - classComunicacion.cs:127-154
----------------------------------------------------------
- Propósito: Procesar una trama de mensaje recibida
- Pasos:
  1. Extrae los bytes 1-4 (longitud del mensaje)
  2. Valida que la longitud sea un número válido (usa int.TryParse, no Convert.ToInt16!)
  3. Verifica que la longitud esté entre 0 y 1019 bytes
  4. Extrae el mensaje (bytes 5 a 5+longitud)
  5. Dispara el evento LlegoMensaje para notificar a Form1
  6. Si hay error, muestra alerta y limpia el buffer

5.6 LimpiarBuffer() - classComunicacion.cs:114-125
------------------------------------------------------
- Propósito: Limpiar el buffer del puerto serial cuando hay datos corruptos
- Función crucial para evitar que el programa se cuelgue por desincronización

6. 4 PREGUNTAS DE EXAMEN (DOCENTE DE REDES) + RESPUESTAS
================================================================================

Pregunta 1:
-----------
¿Por qué es importante usar una bandera (tramaLibre) con el modificador 'volatile'
al enviar mensajes por el puerto serial? Explica qué problema resuelve.

Respuesta 1:
-------------
El modificador 'volatile' es esencial porque:
- Garantiza que todos los hilos lean el valor REAL de la variable desde la memoria principal,
  sin que el compilador la almacene en un registro (caché de CPU).
- Sin 'volatile', dos hilos podrían ver valores diferentes de tramaLibre, lo que causaría
  colisiones al enviar mensajes (dos tramas se enviarían al mismo tiempo, corrompiendo los datos).
- La bandera tramaLibre implementa un mecanismo simple de exclusión mutua para evitar este problema.


Pregunta 2:
-----------
En el código original, se usaba Convert.ToInt16() para convertir la longitud del
mensaje. ¿Por qué es mejor usar int.TryParse() y qué problema de seguridad resuelve?

Respuesta 2:
-------------
- Convert.ToInt16() lanza una excepción si la cadena no es un número válido. Esto puede
  hacer que el programa se cuelgue o crashee si hay desincronización y la trama está corrupta.
- int.TryParse() es seguro: devuelve true/false indicando si la conversión fue exitosa, sin
  lanzar excepciones.
- Esto soluciona el problema de que el programa se cuelgue cuando recibe tramas corruptas o
  desincronizadas, como se mencionó en los videos del problema.


Pregunta 3:
-----------
¿Por qué la trama tiene un tamaño fijo de 1024 bytes? ¿Qué ventajas ofrece esto
en comunicación serial?

Respuesta 3:
-------------
Ventajas del tamaño fijo:
1. **Sincronización fácil**: El receptor sabe exactamente cuántos bytes leer para una trama completa.
2. **Simplicidad**: No es necesario implementar protocolos complejos para detectar el fin de trama.
3. **Predecibilidad**: El buffer de recepción se puede dimensionar de forma fija (1024 bytes), evitando desbordamientos.
4. **Control de flujo**: Al enviar tramas de tamaño fijo, es más fácil gestionar el buffer del puerto.

La trama se compone de:
- 1 byte (tipo) + 4 bytes (longitud) + 1019 bytes (datos+padding) = 1024 bytes.


Pregunta 4:
-----------
¿Por qué se usa un hilo separado para enviar y recibir mensajes (enviandoMensaje y
RecibiendoMensaje) en lugar de hacerlo directamente en el hilo principal de la UI?

Respuesta 4:
-------------
Si hiciéramos la comunicación serial en el hilo principal de la UI:
- La interfaz se "congelaría" (se volvería no responsiva) mientras se envía o recibe un mensaje,
  especialmente si el mensaje es largo o el puerto es lento.
- El usuario no podría interactuar con la aplicación durante ese tiempo.

Al usar hilos separados:
- La UI sigue siendo responsiva en todo momento.
- Las operaciones de envío/recepción no bloquean la interacción del usuario.
- Es una buena práctica de programación para aplicaciones con interfaces gráficas.

7. SOLUCIÓN PARA ENVIAR MENSAJES DE CUALQUIER LONGITUD
================================================================================

7.1 PROBLEMA ANTES DE LA SOLUCIÓN:
-----------------------------------
- El programa solo podía enviar mensajes de hasta 1019 bytes (la carga útil de una trama).
- Si el mensaje era más largo, se cortaba y solo se enviaba la primera parte.
- Si el mensaje era un múltiplo exacto de 1019 bytes, el último trozo era de 1019 bytes y el receptor nunca sabía que el mensaje había terminado (porque solo entregaba el mensaje cuando un trozo era menor a 1019 bytes).

7.2 NUEVAS VARIABLES AGREGADAS:
----------------------------------
- `colaMensajesEnvio` (Queue<byte[]>): Cola FIFO para almacenar los trozos del mensaje antes de enviarlos.
- `mensajeRecibidoCompleto` (StringBuilder): Buffer para ir acumulando los trozos del mensaje en el receptor hasta que llegue el último.

7.3 CÓMO FUNCIONA LA SOLUCIÓN:
---------------------------------

A) ENVÍO (enviarMensaje() y enviandoMensaje()):
1. Convertimos el mensaje completo a bytes UTF-8.
2. Dividimos estos bytes en trozos de máximo 1019 bytes cada uno.
3. Añadimos cada trozo a la cola `colaMensajesEnvio`.
4. **Importante**: Añadimos un trozo VACÍO (longitud 0) al final de la cola. Este es nuestro "marcador de fin de mensaje".
5. Iniciamos el hilo `enviandoMensaje()` si no está corriendo.
6. El hilo `enviandoMensaje()` extrae los trozos de la cola y los envía uno por uno, con un pequeño retardo (50 ms) entre tramas para evitar desincronización.

B) RECEPCIÓN (RecibiendoMensaje()):
1. Cuando llega una trama, extraemos el trozo del mensaje.
2. Si la longitud del trozo es MAYOR que 0, lo añadimos a `mensajeRecibidoCompleto`.
3. Si la longitud del trozo es MENOR que 1019 (incluyendo el trozo vacío de longitud 0):
   a. Convertimos `mensajeRecibidoCompleto` a string (el mensaje completo).
   b. Limpiamos `mensajeRecibidoCompleto` para el próximo mensaje.
   c. Disparamos el evento `LlegoMensaje` para notificar a Form1 que llegó un mensaje completo.

7.4 VENTAJAS DE ESTA SOLUCIÓN:
---------------------------------
1. **Sin límite de longitud**: Se pueden enviar mensajes de cualquier tamaño.
2. **Compatible con el protocolo original**: No modificamos la estructura de la trama (sigue siendo 1024 bytes).
3. **Robusto**: El trozo vacío de fin de mensaje asegura que el receptor siempre sepa cuándo termina, incluso si el último trozo útil es exactamente 1019 bytes.
4. **Orden garantizado**: La cola FIFO asegura que los trozos se envían y reciben en el orden correcto.
5. **UI Responsiva**: Todo el envío y recepción sigue ocurriendo en hilos separados, así que la interfaz no se congela.
